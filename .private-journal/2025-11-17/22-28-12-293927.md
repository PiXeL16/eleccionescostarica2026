## Project Notes

## Mobile Overflow Analysis - Elecciones 2026

Found the 10px mobile overflow issue! The culprit is in the layout hierarchy:

### Root Cause
1. **layout.tsx main container** (line 153): `<div className="container mx-auto px-6 py-8">`
   - Uses `container mx-auto` which is Tailwind's standard responsive container
   - Uses `px-6` = 24px padding on each side (48px total horizontal padding)
   - This wraps ALL content including the comparar page

2. **ComparisonView wrapper** (line 111): `<div className="mx-auto max-w-6xl space-y-8">`
   - Uses `max-w-6xl` (1152px) which is narrower than full width
   - Uses `mx-auto` for centering
   - NO horizontal padding of its own

3. **PartySelector grid** (line 66): `<div className="grid grid-cols-2 gap-3 lg:grid-cols-3">`
   - 2 columns on mobile (grid-cols-2)
   - gap-3 = 12px gap between items
   - No padding constraint

### The Problem Chain
- Mobile viewport: ~375-390px width typical
- Layout outer padding: 24px left + 24px right = 48px total
- Available width for content: 375 - 48 = 327px
- PartySelector items: 2 columns with gap-3
- Each item needs to fit in ~157.5px
- When items have padding (p-4 = 16px), images, and text, they can exceed container
- The comparison cards (p-6 = 24px) may also cause issues

### Affected Components:
1. **PartySelector.tsx** - grid items with p-4
2. **ComparisonView.tsx** - sticky header (line 177) with pb-4, category cards with p-6
3. **Layout.tsx** - main container with px-6

### Solution Paths:
1. Reduce horizontal padding on mobile (e.g., px-4 instead of px-6 on small screens)
2. Use responsive padding: `px-4 md:px-6`
3. Add mobile-specific breakpoints to PartySelector grid
4. Consider adjusting container max-width on mobile

All are safe refactors within the layout styling.
